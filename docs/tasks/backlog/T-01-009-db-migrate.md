# T-01-009 db/migrate.ts — Migration runner and initial base migration

## Goal
Create the migration infrastructure at `src/db/migrate.ts` and generate+run the initial migration (001_base_schema) that creates Symbol, SymbolState, CommonCode tables with pgvector extension enabled.

## Why
The database must be set up before any application logic can read/write data. A proper migration runner ensures repeatable, idempotent schema changes. The initial migration creates the foundational Master (Symbol, SymbolState) and Reference (CommonCode) tables that every subsequent module depends on.

## Inputs
- `src/db/schema.ts` (T-01-008) — Drizzle schema definitions
- `src/db/pool.ts` (T-01-007) — database connection
- `docs/DATA_MODEL.md` — pgvector extension requirement
- `docs/TECH_STACK.md` — drizzle-kit 0.31.10

## Dependencies
T-01-007 (db/pool.ts — migration runner needs database connection)
T-01-008 (db/schema.ts — schema must be defined before migration generation)

## Expected Outputs
- `src/db/migrate.ts` — migration runner script
- `drizzle/` directory with generated migration SQL files
- `drizzle.config.ts` — Drizzle Kit configuration
- Working `bun run migrate` command
- Database with Symbol, SymbolState, CommonCode tables + pgvector extension

## Deliverables
- `src/db/migrate.ts`
- `drizzle.config.ts`
- `drizzle/` migration files (generated by drizzle-kit)

## Constraints
- Migration must be idempotent (safe to run multiple times)
- pgvector extension must be enabled: `CREATE EXTENSION IF NOT EXISTS vector`
- Must use Drizzle Kit's migration system (not raw SQL migration files)
- Migration runner must read DATABASE_URL from environment
- Must handle "already migrated" state gracefully
- `bun run migrate` script in package.json must invoke this

## Steps
1. Create `drizzle.config.ts` at project root:
   - Schema path: `src/db/schema.ts`
   - Output directory: `drizzle/`
   - Driver: `postgres-js`
   - DATABASE_URL from env
2. Generate initial migration: `bunx drizzle-kit generate`
3. Review generated SQL — verify it matches DATA_MODEL.md:
   - CREATE EXTENSION IF NOT EXISTS vector
   - CREATE TABLE symbol (composite PK)
   - CREATE TABLE symbol_state (UUID PK, FK, UNIQUE)
   - CREATE TABLE common_code (composite PK)
   - All CHECK constraints
   - All indexes
4. Create `src/db/migrate.ts`:
   - Import pool and Drizzle migrator
   - Enable pgvector extension before running migrations
   - Run Drizzle migrations from `drizzle/` directory
   - Log migration status
   - Close pool after completion
5. Update `package.json` script: `"migrate": "bun src/db/migrate.ts"`
6. Test migration against a clean database
7. Test migration idempotency (run twice)
8. Verify `bun run typecheck` passes

## Acceptance Criteria
- `bun run migrate` creates all 3 tables successfully on clean DB
- pgvector extension is enabled (verified by `SELECT * FROM pg_extension WHERE extname='vector'`)
- Symbol table exists with correct composite PK
- SymbolState table exists with FK to Symbol, CASCADE delete
- CommonCode table exists with correct composite PK
- Running `bun run migrate` twice completes without errors (idempotent)
- All CHECK constraints are present in the database
- `bun run typecheck` passes

## Test Scenarios
- Running migrate on empty database → creates 3 tables and pgvector extension
- Running migrate twice → second run completes without errors (idempotent)
- Symbol table has composite PK (symbol, exchange) → verified via SQL query
- SymbolState has FK referencing Symbol with CASCADE → delete Symbol row cascades to SymbolState
- CommonCode has composite PK (group_code, code) → verified via SQL query
- SymbolState.fsm_state rejects invalid value → CHECK constraint violation
- Monetary columns (daily_open, etc.) are numeric type → verified via information_schema

## Validation
```bash
bun run typecheck
bun run migrate
bun test --grep "db/migrate"
```
