# T-16-013 Trade Journals Extension Migration

## Goal
Extend the `trade_journals` table with macro context and retrospective columns to support economic context capture and post-trade retrospective analysis.

## Why
Journal entries need to store macro economic context at the time of trade entry/exit and retrospective analysis data generated after the trade closes. These columns are prerequisites for the journal-worker macro enrichment pipeline and the retrospective-worker report generation.

## Inputs
- `packages/shared/db/schema/` — existing DrizzleORM schema definitions
- `trade_journals` table schema (T-16-001)
- `docs/exec-plans/16-macro-retrospective.md` — EP16 macro pipeline spec

## Dependencies
- T-16-001 (trade_journals table — base schema)
- T-13-001 (DrizzleORM migration infrastructure)

## Expected Outputs
- Updated schema file adding three new columns to `trade_journals`
- DrizzleORM migration file generated by `bun run db:generate`
- Test file verifying migration applies cleanly and columns are accessible via ORM

## Deliverables
- Updated `packages/shared/db/schema/trade-journals.ts`:
  - `entry_macro_context JSONB` — macro economic snapshot at trade entry time (nullable)
  - `retrospective_report TEXT` — LLM-generated retrospective analysis text (nullable)
  - `retrospective_generated_at TIMESTAMPTZ` — timestamp when retrospective was produced (nullable)
- Migration SQL file under `packages/shared/db/migrations/`
- `packages/shared/db/__tests__/trade-journals-migration.test.ts`:
  - Verifies new columns exist with correct types
  - Verifies existing rows accept null values for all three columns
  - Verifies JSONB column accepts and round-trips a sample macro context object

## Constraints
- All three columns must be nullable — no DEFAULT value required for backfill compatibility
- `entry_macro_context` schema shape is not enforced at DB level (JSONB is free-form)
- Do not alter existing columns or their constraints
- Migration must be idempotent (DrizzleORM handles this via migration tracking)
- Do not import Elysia, CCXT, or Slack in schema files

## Steps
1. Write failing tests first (RED):
   - Test: `trade_journals` schema includes `entry_macro_context` column of type JSONB
   - Test: `trade_journals` schema includes `retrospective_report` column of type TEXT
   - Test: `trade_journals` schema includes `retrospective_generated_at` column of type TIMESTAMPTZ
   - Test: inserting a row without the new columns succeeds (existing rows remain valid)
   - Test: JSONB column round-trips a sample `{ indicators: {}, events: [] }` object
2. Add column definitions to `packages/shared/db/schema/trade-journals.ts` (GREEN)
3. Run `bun run db:generate` to produce the migration file
4. Verify migration file contains the three `ALTER TABLE ADD COLUMN` statements
5. Run tests to confirm GREEN
6. Refactor (REFACTOR): ensure column names follow existing snake_case convention in schema

## Acceptance Criteria
- `trade_journals` schema exports `entryMacroContext`, `retrospectiveReport`, `retrospectiveGeneratedAt` fields
- Generated migration SQL contains `ADD COLUMN entry_macro_context jsonb`, `ADD COLUMN retrospective_report text`, `ADD COLUMN retrospective_generated_at timestamptz`
- Existing rows accept NULL for all three columns (backfill not required)
- `bun test -- --filter "journal.*migration"` passes
- `bun run typecheck` passes
- `bun run lint` passes

## Validation
```bash
bun run db:generate
bun test -- --filter "journal.*migration"
bun run typecheck
bun run lint
```

## Out of Scope
- Populating `entry_macro_context` values (T-16-014)
- Generating retrospective reports (T-16-008)
- Indexing the JSONB column for query performance
- Adding `exit_macro_context` (not required by current spec)
