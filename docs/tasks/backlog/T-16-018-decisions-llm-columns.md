# T-16-018 Decisions LLM Columns

## Goal
Extend the `decisions` table with LLM evaluation result columns so that LLM filter outcomes can be stored alongside kNN-derived decisions for audit, analysis, and override tracking.

## Why
When the LLM filter overrides a kNN decision, the system must record both the original kNN recommendation and the LLM evaluation result. Without dedicated columns, audit trails are lost and operators cannot diagnose why an alert was suppressed or why a trade was not placed. These columns are required before the LLM decision routing pipeline (T-16-019) can persist its output.

## Inputs
- `packages/shared/db/schema/decisions.ts` — existing decisions schema
- LLM decision filter service contract (T-16-010) — defines the shape of the evaluation result
- `docs/exec-plans/16-macro-retrospective.md` — EP16 LLM columns spec

## Dependencies
- T-16-017 (strategies.use_llm_filter — establishes the feature flag that gates these columns being populated)

## Expected Outputs
- Updated `decisions` schema with five new nullable LLM columns
- DrizzleORM migration file
- Tests verifying schema and nullable behaviour

## Deliverables
- Updated `packages/shared/db/schema/decisions.ts`:
  - `llm_action TEXT` — LLM decision output: `'LONG'`, `'SHORT'`, or `'PASS'` (nullable)
  - `llm_reason TEXT` — free-text explanation from LLM response (nullable)
  - `llm_confidence REAL` — confidence score 0.0–1.0 from LLM response (nullable)
  - `llm_risk_factors JSONB` — structured risk factor list from LLM response (nullable)
  - `llm_evaluated_at TIMESTAMPTZ` — timestamp when LLM evaluation completed (nullable)
- Migration SQL file under `packages/shared/db/migrations/`
- `packages/shared/db/__tests__/decisions-llm-columns.test.ts`:
  - Test: all five columns exist with correct types
  - Test: all five columns are nullable (existing rows unaffected)
  - Test: a decision row can be inserted without LLM columns (kNN-only path)
  - Test: a decision row can be updated with LLM evaluation results
  - Test: `llm_risk_factors` JSONB accepts and round-trips a sample risk factors array
  - Test: `llm_confidence` accepts values in range 0.0–1.0 (spot-check boundary values)

## Constraints
- All five columns must be nullable — kNN-only decisions never populate them
- `llm_action` is free-text at DB level — validation of allowed values happens in application code
- `llm_confidence` is REAL (single precision float) at DB level — application code uses Decimal.js for comparisons
- Do not alter existing columns or their constraints
- Migration must be generated by DrizzleORM, not written by hand
- Do not import Elysia, CCXT, or Slack in schema files

## Steps
1. Write failing tests first (RED):
   - Test: each of the five columns is present in schema with correct DrizzleORM type
   - Test: insert decision without LLM columns succeeds
   - Test: update decision with LLM evaluation data succeeds
   - Test: JSONB round-trip for `llm_risk_factors`
2. Add five column definitions to `packages/shared/db/schema/decisions.ts` (GREEN)
3. Run `bun run db:generate` to produce the migration file
4. Verify migration SQL contains five `ADD COLUMN` statements
5. Run tests to confirm GREEN
6. Refactor (REFACTOR): group LLM columns with a comment block `// LLM evaluation fields` in schema file

## Acceptance Criteria
- `decisions` schema exports `llmAction`, `llmReason`, `llmConfidence`, `llmRiskFactors`, `llmEvaluatedAt` fields
- All five fields are nullable in the generated migration SQL
- Existing decision rows accept NULL for all five columns
- `llm_risk_factors` JSONB round-trips correctly
- `bun test -- --filter "decision.*llm"` passes
- `bun run typecheck` passes
- `bun run lint` passes

## Validation
```bash
bun run db:generate
bun test -- --filter "decision.*llm"
bun run typecheck
bun run lint
```

## Out of Scope
- LLM evaluation logic (T-16-010)
- Routing decisions through LLM filter (T-16-019)
- Index on `llm_action` for query performance
- `llm_action` enum constraint at DB level
- `llm_confidence` range check constraint at DB level
