## What was hard

### 1. DB test parallelism failures (~15 min added to Wave 3)

**What happened**: Wave 3 merged T-04-003 (repository) and T-04-005 (cleanup), both using test-db. When `bun test` ran all test files in parallel, multiple files attempted to use the same DB pool simultaneously, causing deadlocks and failures.

**Root cause**: Individual test files called `closeTestDb()` in `afterAll`, which closed the shared pool while other test files were still using it.

**Fixes applied** (commit cf06c5d):
1. Removed `closeTestDb()` from individual afterAll hooks
2. Added `tests/helpers/global-teardown.ts` (9 LOC) — single pool close at process exit
3. `pool.test.ts` needed `beforeEach(closePool)` for clean state (it tests the pool itself)
4. `test-db.test.ts` had wrong schema reference for drizzle migrations table

**Time cost**: ~15 min added to Wave 3 (32 min wall time vs expected ~4 min).

**Systemic lesson**: Test isolation at the DB level requires explicit design. This is not a Bun-specific issue; any test runner with parallel execution and shared resources faces the same problem. See pattern file: `docs/patterns/db-test-parallelism.md`.

### 2. Worktree merge friction

Parallel agents used git worktrees but did not commit their changes. This meant files had to be manually copied back to the main worktree before committing.

**Impact**: Minor time cost per wave, but error-prone. A forgotten file would cause test failures in the merge commit.

**Root cause**: The worktree workflow assumed agents would commit, but the harness implementation copies files without git operations.

### 3. Pre-existing lint issue (biome nested config)

Worktrees from previous executions left behind nested `biome.json` config files, causing biome to fail with conflicting configuration errors during lint checks.

**Impact**: Required manual cleanup before lint could pass.

**Root cause**: Stale worktree artifacts were not cleaned up after previous sessions.

## Patterns discovered

### [New] DB test parallelism requires global teardown
- **Observed in**: Wave 3 merge (T-04-003 + T-04-005)
- **Category**: reliability
- **Description**: Multiple test files sharing a DB pool will deadlock if individual files close the pool in afterAll
- **Root cause**: Bun test runner executes files in parallel; shared singleton pool cannot be closed per-file
- **Recommendation**: Use global-teardown.ts for pool cleanup; individual files only TRUNCATE tables in beforeEach
- **Pattern file**: `docs/patterns/db-test-parallelism.md`

### [New] Pre-implementation review prevents rework
- **Observed in**: EP-04 task review phase
- **Category**: efficiency
- **Description**: Reviewing task definitions before coding found 3 Critical issues that would have caused rework
- **Root cause**: Task generation is imperfect; cross-task dependency and responsibility analysis requires a dedicated review pass
- **Recommendation**: Always run pre-implementation review after task generation. Budget ~15 min per epic
- **Pattern file**: `docs/patterns/pre-implementation-review.md`

### [New] Constructor injection for testability
- **Observed in**: T-04-004, T-04-009, T-04-010
- **Category**: architecture
- **Description**: Modules accept dependencies via constructor/options objects instead of importing singletons
- **Root cause**: DB-dependent modules need both mock (unit test) and real DB (integration test) modes
- **Recommendation**: All L3+ modules should accept their L1/L2 dependencies via injection
- **Pattern file**: `docs/patterns/constructor-injection.md`

### [Confirmed] Scaffold task consolidation
- **First observed**: EP-03 (T-03-007/008/009)
- **EP-04 status**: Not applicable (no scaffold tasks in EP-04). Rule remains valid for future epics with repetitive patterns.

### [Confirmed] Interface changes in first task

## Recommendations

### 1. Codify global-teardown pattern for all future DB test files
Every test file that uses test-db must NOT call closeTestDb() in afterAll. Biome lint 규칙 또는 코드 리뷰로 이를 강제한다. The global-teardown.ts pattern is the only safe way to close the shared pool.

### 2. Consolidate TIMEFRAME_DURATION_MS
`collector.ts` and `gap-detection.ts` both define timeframe-to-milliseconds mappings independently. Extract to a shared constant in `src/core/constants.ts` to prevent drift. Already flagged in anti-patterns.md but not yet resolved.

### 3. Run pre-implementation review as a standard stage
EP-04 proved the value of reviewing task definitions before implementation. Add this as a mandatory step in the harness workflow between task generation (stage 7) and implementation (stage 8). Cost: ~15 min. Expected saving: 30+ min in avoided rework.

### 4. Automate worktree cleanup
Stale worktree artifacts (biome.json, uncommitted files) caused friction in EP-03 and EP-04. Add a cleanup script that removes worktree-generated files before each session starts.

### 5. Establish steady-state velocity baseline
With 2 epics of data, propose: **steady-state target = 5 min/task** for DB-heavy epics, **3 min/task** for pure logic epics. Track in future retros.
