# T-179 Multiuser DB schema

## Goal
Add `user_id UUID NOT NULL` foreign key columns (referencing `user.id` from better-auth) to the six tables that require direct user isolation: `strategies`, `orders`, `kill_switch_state`, `kill_switch_events`, `daily_loss_limits`, `daily_pnl_tracking`. Generate and apply the migration including a backfill SQL that assigns the initial admin user's id to all existing rows.

## Why
Currently all core tables are user-unaware — any authenticated user can read or modify any record. This is unsafe even in a single-user deployment because the system is exposed to the internet. Adding `user_id` as a NOT NULL FK enforces ownership at the database level, making cross-user data leaks structurally impossible.

The tables `strategy_events`, `decisions`, `alerts`, `entry_snapshots`, and `vector_table_registry` are intentionally excluded: they inherit isolation through their `strategy_id` FK once strategies are isolated.

### Migration backfill SQL (embed in migration or run separately)
```sql
-- Assign all existing rows to the first user (initial admin)
UPDATE strategies         SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
UPDATE orders             SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
UPDATE kill_switch_state  SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
UPDATE kill_switch_events SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
UPDATE daily_loss_limits  SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
UPDATE daily_pnl_tracking SET user_id = (SELECT id FROM "user" LIMIT 1) WHERE user_id IS NULL;
```

### Index
Add an index on `user_id` for each modified table to avoid full table scans on user-filtered queries:
```typescript
index("strategies_user_id_idx").on(table.userId)
```

## Constraints
- `user_id` column must be `NOT NULL` — nullable user_id would defeat the isolation invariant
- FK must reference `user.id` from the better-auth schema, not the legacy `users.id`
- The backfill SQL must run before the NOT NULL constraint is finalized (or use a default + backfill + drop default pattern if Drizzle requires it)
- Running DB is required for `bun run db:migrate`. Schema generation and typecheck can be validated without DB.
- Do NOT add `user_id` to: `strategy_events`, `decisions`, `alerts`, `entry_snapshots`, `vector_table_registry` — these are out of scope by design
- Do NOT modify `db/schema/users.ts` (legacy; migration aligns it to better-auth's `user` table in T-176)

## Implementation Notes

**Date**: 2026-03-23

### Key decision: `text` type for userId column

`authUser.id` in `db/schema/better-auth.ts` is `text("id").primaryKey()` — not `uuid`. better-auth uses text PKs so it can inject IDs via `generateId` without driver coercion. All six `userId` columns therefore use `text("user_id")` to match the FK target type exactly.

### Test approach

Tests inspect Drizzle's runtime object structure:
- Column presence/name/notNull: via direct property access (`table.userId.name`, `.notNull`)
- FK verification: via `Symbol(drizzle:PgInlineForeignKeys)` which holds `{ reference() }` objects; validates `user_id → user.id`
- Index verification: via `Symbol(drizzle:ExtraConfigBuilder).toString()` which contains the declared index name
