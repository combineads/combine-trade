# 백테스트

## 5.1 백테스트란?

백테스트는 **과거 시장 데이터를 사용해 트레이딩 전략을 시뮬레이션**하는 과정입니다.
실제 돈을 투입하기 전에 전략이 수익을 낼 수 있는지 검증합니다.

예를 들어, "2023년 1월부터 2025년 12월까지 BTCUSDT에 이 전략을 적용했다면
어떤 결과가 나왔을까?"를 계산하는 것입니다.

### 백테스트의 한계

백테스트 결과가 좋다고 해서 미래에도 반드시 수익이 나는 것은 아닙니다.
가장 큰 위험은 **과적합(Overfitting)**입니다:

> **과적합이란?**
> 과거 데이터에 지나치게 맞춰진 전략은 "과거의 정답지를 외운 것"과 같습니다.
> 시험 문제(미래 시장)가 바뀌면 성적이 급락합니다.
>
> 예시: KNN 이웃 수(top_k)를 50으로 했을 때보다 47로 했을 때 과거 3년 수익이
> 미세하게 더 높다고 47을 선택하면, 그건 과거 노이즈에 맞춘 것일 뿐
> 미래에도 47이 더 좋다는 보장이 없습니다.

이 과적합 문제를 해결하기 위해 **WFO(Walk-Forward Optimization)**를 사용합니다.

## 5.2 CLI 사용법

### 단순 백테스트

```bash
# 기본 사용법
bun run backtest -- --symbol BTCUSDT --start 2024-01-01 --end 2024-06-01

# 거래소 지정
bun run backtest -- --symbol BTCUSDT --exchange okx --start 2024-01-01 --end 2024-06-01
```

### WFO 모드

```bash
# WFO 실행 (3년 데이터 권장)
bun run backtest -- --mode wfo --symbol BTCUSDT --start 2022-01-01 --end 2025-01-01

# 병렬 실행 (스레드 수 지정)
bun run backtest -- --mode wfo --symbol BTCUSDT --start 2022-01-01 --end 2025-01-01 --threads 4
```

### CLI 옵션

| 옵션 | 필수 | 기본값 | 설명 |
|------|------|--------|------|
| `--symbol` | O | — | 거래 심볼 (BTCUSDT, XAUTUSDT) |
| `--exchange` | X | binance | 거래소 |
| `--start` | O | — | 시작 날짜 (YYYY-MM-DD) |
| `--end` | O | — | 종료 날짜 (YYYY-MM-DD) |
| `--mode` | X | backtest | `backtest` 또는 `wfo` |
| `--threads` | X | CPU/2 | 병렬 스레드 수 |

> **EP-19**: 백테스트 및 WFO 집계 결과는 자동으로 DB에 저장됩니다 (`backtest_runs` 테이블).
> 개별 거래 기록은 여전히 메모리에만 수집되며, 저장되는 것은 집계 지표(기대값, MDD, 승률 등)입니다.

## 5.3 백테스트 결과 해석

백테스트가 완료되면 아래 지표가 출력됩니다:

| 지표 | 의미 | 좋은 결과 | 나쁜 결과 |
|------|------|----------|----------|
| **총 거래 (Total trades)** | 전체 거래 횟수 | 30건 이상 (통계적 유의성) | 10건 미만 (신뢰 불가) |
| **승률 (Win rate)** | 이긴 거래 비율 | 50% 이상 | 30% 미만 |
| **기대값 (Expectancy)** | 거래 1건당 평균 수익 | **양수 (필수)** | 음수 = 손실 전략 |
| **MDD (Max Drawdown)** | 최고점 대비 최대 하락폭 | -10% 이내 | -20% 초과 |
| **프로핏 팩터 (Profit Factor)** | 총 수익 / 총 손실 | 1.5 이상 | 1.0 미만 = 손실 |
| **샤프 비율 (Sharpe Ratio)** | 위험 대비 수익률 | 1.0 이상 | 0 미만 |

### 지표 상세 설명

**기대값 (Expectancy)**

거래 1건을 실행했을 때 평균적으로 얼마를 벌 수 있는지를 나타냅니다.

```
기대값 = (평균 수익 × 승률) - (평균 손실 × 패률)
```

예시: 평균 수익 100 USDT, 승률 60%, 평균 손실 80 USDT, 패률 40%일 때
→ 기대값 = (100 × 0.6) - (80 × 0.4) = 60 - 32 = **28 USDT**
→ 거래 1건당 평균 28 USDT 수익 기대

> 기대값이 양수여야 전략이 장기적으로 수익을 냅니다.
> **기대값 > 0은 모든 검증의 최소 조건**입니다.

**MDD (Maximum Drawdown, 최대 낙폭)**

자산이 최고점에서 최저점까지 얼마나 떨어졌는지를 나타냅니다.

예시: 계좌가 10,000 → 12,000 → 9,600으로 변했다면
→ MDD = (12,000 - 9,600) / 12,000 = **-20%**

> MDD는 "최악의 경우 얼마나 잃을 수 있는가"를 보여줍니다.
> MDD -10%이면 1,000만원 계좌에서 최대 100만원 손실 구간이 있었다는 의미입니다.

**프로핏 팩터 (Profit Factor)**

```
프로핏 팩터 = 총 수익 합계 / 총 손실 합계 (절대값)
```

- 1.0 = 본전 (벌어도 다 잃음)
- 2.0 = 잃은 것의 2배를 벌었음
- 1.0 미만 = 총 손실이 총 수익보다 큼

**샤프 비율 (Sharpe Ratio)**

수익률의 변동성(위험) 대비 얼마나 수익을 냈는지를 나타냅니다.

- 1.0 이상 = 위험 대비 괜찮은 수익
- 2.0 이상 = 매우 좋음
- 0 이하 = 수익이 없거나 손실

> 동일한 수익이라도 변동이 적을수록 샤프 비율이 높습니다.
> "꾸준히 조금씩 버는 전략"이 "크게 벌었다 크게 잃는 전략"보다 샤프가 높습니다.

## 5.4 WFO (Walk-Forward Optimization) 상세

### WFO가 필요한 이유

일반 백테스트의 문제점:

```
[====== 3년 전체 데이터로 파라미터 최적화 ======]
         ↓
   과거에 최적인 파라미터를 찾았지만,
   이게 미래에도 통할지 검증할 방법이 없음
         ↓
   → 과적합 위험이 높음
```

WFO의 해결 방법:

```
[== 학습 기간 ==][= 검증 =]
   파라미터 최적화     찾은 파라미터로 테스트
   (과거 데이터)       (미래 데이터)
         ↓
   학습 데이터로 찾은 파라미터가
   처음 보는 데이터에서도 통하는지 확인
         ↓
   → 과적합 여부를 객관적으로 판단 가능
```

### WFO 작동 원리

WFO는 데이터를 **학습(In-Sample)** 구간과 **검증(Out-of-Sample)** 구간으로 나누어,
학습 구간에서 찾은 최적 파라미터가 검증 구간에서도 유효한지 반복 확인합니다.

기본 설정(IS=6개월, OOS=2개월, 롤링=1개월)으로 3년 데이터를 사용하면:

```
데이터: 2022-01 ─────────────────────────────────────── 2025-01

윈도우 0:  [===== IS 6개월 =====][= OOS 2개월 =]
           2022-01              2022-07        2022-09

윈도우 1:     [===== IS 6개월 =====][= OOS 2개월 =]
              2022-02              2022-08        2022-10

윈도우 2:        [===== IS 6개월 =====][= OOS 2개월 =]
                 2022-03              2022-09        2022-11

  ...        (1개월씩 앞으로 이동하며 반복)

윈도우 N:                          [===== IS 6개월 =====][= OOS 2개월 =]
                                   2024-05              2024-11        2025-01
```

### 각 윈도우에서 일어나는 일

**1단계: 학습 (In-Sample)**

IS 구간의 데이터로 다양한 파라미터 조합을 테스트합니다.

```
파라미터 조합 예시:
  KNN top_k = 30, 40, 50, 60, 70, 80, 90, 100  (8가지)
  max_pyramid = 1, 2, 3                         (3가지)
  max_daily_loss = 5%, 7.5%, 10%, 12.5%, 15%    (5가지)
  → 총 8 × 3 × 5 = 120가지 조합

각 조합으로 IS 구간 백테스트 실행
→ 기대값이 가장 높은 조합을 "최적 파라미터"로 선택
```

**2단계: 검증 (Out-of-Sample)**

IS에서 찾은 최적 파라미터로 OOS 구간 백테스트를 실행합니다.
OOS 데이터는 IS 최적화에 사용되지 않은 **새로운 데이터**입니다.

```
IS 최적 파라미터: top_k=50, max_pyramid=2, max_daily_loss=10%
→ 이 파라미터로 OOS 구간 백테스트 실행
→ OOS 기대값 산출
```

**3단계: 효율성 계산**

```
WFO 효율성 = OOS 기대값 / IS 기대값
```

| IS 기대값 | OOS 기대값 | 효율성 | 해석 |
|----------|----------|--------|------|
| 2.0 | 1.5 | 0.75 | 매우 좋음 — 검증에서도 75% 유지 |
| 2.0 | 1.0 | 0.50 | 통과 — 최소 기준 충족 |
| 2.0 | 0.5 | 0.25 | 과적합 의심 — 검증 성과 급락 |
| 2.0 | -0.3 | 음수 | 과적합 확정 — 실전에서 손실 |
| -0.5 | — | 건너뜀 | IS부터 수익이 안 나면 해당 윈도우 제외 |

### 최종 결과

모든 윈도우의 효율성을 평균내어 **전체 WFO 효율성**을 계산합니다.

```
전체 WFO 효율성 = 유효 윈도우들의 효율성 평균

예시:
  윈도우 0: 효율성 0.72
  윈도우 1: 효율성 0.65
  윈도우 2: 효율성 0.58
  윈도우 3: IS 기대값 < 0 (건너뜀)
  윈도우 4: 효율성 0.81
  윈도우 5: 효율성 0.49
  ─────────────────────────
  전체 효율성 = (0.72 + 0.65 + 0.58 + 0.81 + 0.49) / 5 = 0.65
```

### WFO 성공 기준

| 기준 | 조건 | 의미 |
|------|------|------|
| OOS 기대값 | > 0 (양수) | 검증 구간에서도 수익이 남 |
| WFO 효율성 | **> 0.5** | IS 성과의 50% 이상이 OOS에서 유지됨 |
| MDD | 허용 범위 이내 | 낙폭이 감당 가능한 수준 |

> **EP-19 자동 게이트**: WFO 효율성 > 0.5 기준은 시스템이 자동으로 강제합니다.
> 기준 미달 시 CommonCode에 `WFO_GATE=FAIL`이 기록되고 `live` 모드 진입이 차단됩니다.
> 이 게이트는 수동으로 우회할 수 없습니다. WFO 통과 시 CommonCode가 자동 업데이트됩니다.

> **효율성 > 0.5의 의미:**
> 학습 데이터에서 기대값이 2.0이었다면, 새로운 데이터에서도 최소 1.0 이상이어야 합니다.
> 이 기준을 충족하면 전략이 과적합이 아닌 **실제 시장 패턴을 포착**하고 있다고 판단합니다.

### WFO 결과 해석 가이드

| 결과 | 의미 | 조치 |
|------|------|------|
| 효율성 > 0.7 | 매우 강건한 전략 | 다음 단계(analysis 모드)로 진행 |
| 효율성 0.5~0.7 | 합격 — 적절한 강건성 | 진행 가능, 리스크 보수적 설정 권장 |
| 효율성 0.3~0.5 | 경계 — 과적합 가능성 | 파라미터 단순화 또는 데이터 기간 확대 필요 |
| 효율성 < 0.3 | 과적합 확정 | **라이브 진행 금지**. 전략 자체 재검토 필요 |
| 유효 윈도우 0개 | IS에서도 수익 불가 | 전략이 근본적으로 작동하지 않음 |

## 5.5 파라미터 검색 과정

WFO 각 윈도우의 IS 단계에서는 **2단계 파라미터 검색**이 실행됩니다.

### 1단계: 그리드 검색 (Grid Search)

주요 파라미터의 모든 조합을 체계적으로 테스트합니다:

| 그룹 | 파라미터 | 탐색 범위 | 간격 |
|------|---------|----------|------|
| KNN | top_k (및 모든 KNN.* 파라미터) | 30~100 | 10 |
| SYMBOL_CONFIG | risk_pct | 0.005~0.02 | 0.005 |

> **EP-19 화이트리스트 변경**: 튜닝 가능한 파라미터는 `KNN.*` (전체) 및 `SYMBOL_CONFIG.risk_pct`만 허용됩니다.
> `POSITION`, `LOSS_LIMIT`, `SLIPPAGE`, `TIME_DECAY` 그룹은 튜닝 대상에서 제외되었습니다.
> 이 값들은 구조적 앵커로 취급되어 코드에 고정됩니다.

### 2단계: 랜덤 검색 (Random Search)

1단계에서 상위 5개 조합을 선택한 후, 각각에 대해
`FEATURE_WEIGHT.*` (모든 피처 가중치) 파라미터를 랜덤으로 20회 샘플링합니다.

```
그리드 검색 ~500개 조합
    ↓ 상위 5개 선택
랜덤 검색 5 × 20 = 100개 조합
    ↓
총 ~600개 조합 중 기대값 최고인 파라미터 세트 선택
```

### 검색 대상에서 제외되는 파라미터

**구조적 앵커**는 절대 검색 대상에 포함되지 않습니다:

- BB20 (length=20, stddev=2)
- BB4 (length=4, stddev=4)
- MA 기간 (20, 60, 120)
- 정규화 방법 (min-max)
- 벡터 차원 (202)

이 값들은 전략의 근간이며, 최적화하면 과적합 위험이 매우 높아집니다.

| 직렬 실행 시 | ~20시간 |

| 8스레드 병렬 | ~2.5시간 |

### 출력 예시

```
WFO complete — 22 valid windows (3 skipped), overall efficiency: 0.6520

Window details:
  #0  IS exp: 1.82  OOS exp: 1.31  efficiency: 0.72
  #1  IS exp: 1.65  OOS exp: 1.07  efficiency: 0.65
  #2  IS exp: 1.43  OOS exp: 0.83  efficiency: 0.58
  #3  IS exp: -0.12  (skipped — IS not profitable)

```

### 결과 해석

위 예시에서:
- 25개 윈도우 중 22개가 유효 (IS 기대값 > 0)
- 3개는 IS에서도 수익이 나지 않아 건너뜀
- **전체 효율성 0.65 → 0.5 이상이므로 합격**
- 학습 구간 성과의 65%가 검증 구간에서 유지되었음

## 5.7 백테스트 원칙

### 코드 경로 동일성

백테스트는 라이브와 **동일한 코드 경로**를 사용합니다.
전략 로직에 `if (isBacktest)` 같은 분기가 존재하지 않습니다.

- `PipelineDeps` 의존성 주입(DI)으로 거래소 어댑터만 교체
- 지표 계산, 신호 판단, 포지션 사이징 로직은 100% 동일
- `MockExchangeAdapter`가 실제 `ExchangeAdapter` 인터페이스를 완전 구현

> **왜 이게 중요한가?**
> 백테스트 전용 코드가 따로 있으면, 백테스트에서는 잘 되는데
> 실제 운영에서 다르게 동작할 수 있습니다.
> 코드 경로가 동일하면 "백테스트 수익 = 실제 수익" 신뢰도가 높아집니다.

### 선행 편향(Lookahead Bias) 방지

`MockExchangeAdapter`는 **시간 순서를 강제**합니다:

- `fetchOHLCV`는 `currentTimestamp` 이하의 캔들만 반환
- 미래 데이터에 접근할 수 없음
- 이 규칙은 코드 레벨에서 강제됨

> **선행 편향이란?**
> 백테스트 시 "아직 발생하지 않은 미래 데이터"를 참조하는 오류입니다.
> 예: 오후 2시에 오후 3시 종가를 사용해 매매 판단 → 현실에서는 불가능
> 이 시스템은 코드 수준에서 미래 데이터 접근을 차단하여 이 문제를 방지합니다.

### 데이터 저장 방식

- 개별 거래 기록: **메모리에만** 수집 (DB에 저장하지 않음)
- 집계 결과 (기대값, MDD, 승률 등): DB에 저장
- WFO 결과: 상위 WFO 실행 + 하위 윈도우별 결과가 `parent_id`로 연결되어 저장

## 5.8 WFO 설정 변경

### WFO 윈도우 파라미터

```sql
-- In-sample 기간 변경 (6개월 → 9개월) — 더 많은 학습 데이터
UPDATE common_code SET value = '9'::jsonb
WHERE group_code = 'WFO' AND code = 'in_sample_months';

-- Out-of-sample 기간 변경 (2개월 → 3개월) — 더 긴 검증 기간
UPDATE common_code SET value = '3'::jsonb
WHERE group_code = 'WFO' AND code = 'out_sample_months';

-- 롤링 간격 변경 (1개월 → 2개월) — 윈도우 수 줄이기
UPDATE common_code SET value = '2'::jsonb
WHERE group_code = 'WFO' AND code = 'roll_months';
```

### 파라미터 선택 가이드

| 상황 | 조정 방향 |
|------|----------|
| 효율성이 너무 낮음 (< 0.3) | IS 기간 늘리기 (더 많은 학습 데이터) |
| 유효 윈도우가 너무 적음 | IS 기간 줄이기 또는 파라미터 범위 축소 |
| 실행 시간이 너무 오래 걸림 | 롤링 간격 늘리기, 스레드 수 늘리기 |
| 결과의 분산이 큼 | OOS 기간 늘리기 (더 안정적인 검증) |

## 5.9 캔들 데이터 준비

백테스트에는 충분한 과거 캔들 데이터가 필요합니다.

```bash
# 과거 캔들 데이터 시드
bun scripts/seed.ts
```

### 데이터 요구량

| 모드 | 최소 데이터 | 권장 데이터 |
|------|-----------|-----------|
| 단순 백테스트 | 3개월 + 120봉 선행 | 6개월 이상 |
| WFO | IS + OOS + 120봉 선행 | **3년 이상** |

> 지표 계산을 위해 최소 120봉 이상의 선행 데이터가 필요합니다 (MA120 기준).
> WFO는 다수의 IS/OOS 윈도우를 생성하므로 데이터가 많을수록 신뢰도가 높아집니다.

### 메모리 사용량 참고

3년 1분봉 데이터 기준 (1심볼):

| 항목 | 크기 |
|------|------|
| 캔들 데이터 | ~150MB |
| 벡터 데이터 | ~1.2GB |
| 지표 캐시 | ~50MB |
| 합계 | ~1.4GB |

> 서버 RAM이 4GB 이상인 환경을 권장합니다.

## 5.10 자주 묻는 질문

### 백테스트 기대값이 양수인데 WFO 효율성이 낮으면?

과적합일 가능성이 높습니다. 파라미터 검색 범위를 좁히거나,
더 긴 데이터 기간을 사용해 보세요.

### WFO 효율성이 정확히 0.5이면 통과인가요?

네, > 0.5이 기준이지만 0.5 근처는 경계입니다.
0.6 이상을 목표로 하는 것이 안전합니다.

### 모든 윈도우에서 IS 기대값이 음수이면?

전략 자체가 현재 시장 환경에서 작동하지 않을 수 있습니다.
파라미터 문제가 아니라 전략 로직 재검토가 필요합니다.

### 백테스트와 WFO 중 어떤 걸 먼저 해야 하나요?

1. 먼저 **단순 백테스트**로 전략이 기본적으로 수익이 나는지 확인
2. 수익이 나면 **WFO**로 과적합 여부 검증
3. WFO 통과 후 **analysis 모드**로 실시간 관찰

### WFO 실행 중 중단해도 되나요?

안전하게 중단 가능합니다 (Ctrl+C). 다만 부분 결과는 저장되지 않으므로
처음부터 다시 실행해야 합니다.
