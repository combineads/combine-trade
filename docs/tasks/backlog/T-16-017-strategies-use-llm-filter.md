# T-16-017 Strategies Use LLM Filter Column

## Goal
Add a `use_llm_filter` boolean column to the `strategies` table to enable per-strategy opt-in for LLM-based decision filtering.

## Why
Not every strategy should route decisions through the LLM filter. Lower timeframes (< 15m) are too latency-sensitive for LLM evaluation. Providing a per-strategy flag lets operators enable LLM filtering only for strategies where the latency tradeoff is acceptable. Without this column the vector-worker cannot know which strategies should route through the LLM decision path.

## Inputs
- `packages/shared/db/schema/strategies.ts` — existing strategies schema
- Strategy CRUD API endpoints (EP10 or equivalent)
- Strategy detail API response type
- `docs/exec-plans/16-macro-retrospective.md` — EP16 LLM filter spec

## Dependencies
- T-16-010 (LLM decision filter service — defines the filtering contract this flag gates)

## Expected Outputs
- Updated `strategies` schema with `use_llm_filter` column
- DrizzleORM migration file
- Updated strategy CRUD to accept and return the new field
- Updated strategy detail API response including the field
- Tests covering schema change and API round-trip

## Deliverables
- Updated `packages/shared/db/schema/strategies.ts`:

- Migration SQL file under `packages/shared/db/migrations/`

- Updated strategy detail API response type to include `useLlmFilter: boolean`
- `packages/shared/db/__tests__/strategies-llm-filter.test.ts`:

## Constraints

- Column must be `NOT NULL` with `DEFAULT false`
- Do not rename or remove any existing columns
- Migration must be generated by DrizzleORM (`bun run db:generate`), not written by hand
- Strategy CRUD changes must follow existing AOP/IoC patterns — no manual service instantiation
- Do not import Elysia in schema or CRUD service files
