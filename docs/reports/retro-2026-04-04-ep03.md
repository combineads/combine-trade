## Patterns discovered

### [New] Error mapping centralization
- **Observed in**: T-03-001, T-03-002, T-03-003, T-03-004
- **Category**: efficiency
- **Description**: CCXT 에러를 각 메서드에서 개별 catch 하지 않고, `mapError()`에서 한 번만 매핑. 7가지 도메인 에러 타입(`RateLimitError`, `AuthError`, `InsufficientFundsError`, `OrderNotFoundError`, `NetworkError`, `ExchangeNotImplementedError`, `ExchangeError`)으로 분류.
- **Root cause**: CCXT가 거래소별로 다른 에러를 던지는데, 각 메서드마다 try-catch를 넣으면 중복 코드 폭발.
- **Recommendation**: 이 패턴을 다른 외부 API 통합(경제 캘린더, Slack 등)에도 적용.

### [New] Scaffold-first for multi-implementation interfaces
- **Observed in**: T-03-007, T-03-008, T-03-009
- **Category**: efficiency
- **Description**: ExchangeAdapter 인터페이스의 4개 구현체 중 Phase 1(Binance)만 실제 구현하고, 나머지 3개는 `NotImplementedError` stub으로 즉시 정의. 컴파일 타입 안전을 확보하면서 구현 범위를 제한.
- **Root cause**: 4개 거래소를 한 에픽에서 모두 구현하면 범위 폭발. Stub으로 인터페이스 준수를 보장하면서 점진적 구현.
- **Recommendation**: 멀티 어댑터 패턴(알림 채널, 데이터 소스 등) 설계 시 동일 접근법 적용.
